# Игровой движок. Что собой представляет и какие проблемы решает?
Ответ:

Игровой движок предоставляет готовый набор инструментов и систем для разработки игр.
Он решает задачи:
- рендеринга
- физики
- анимации
- обработки ввода
- звука
- управления сценами
- сборки под разные платформы

Благодаря этому разработчик может сосредоточиться на игровой логике, а не реализовывать базовые технологии с нуля.

# Что такое MonoBehaviour? От чего он наследуется? Можно ли создавать тип наследуемый от Component?

Ответ:

**MonoBehaviour** (Unity Life Sycle или последовательность вызова методов в скриптах Unity) – класс, который предоставляет непосредственно движок Unity и который берет на себя функционал за то, чтобы объекты (gameObject’ы), расположенные на сцене, чтобы о них движок знал.

Он наследуется от Behaviour (а значит, и от Component). То есть, то, как объект создается, как он живет и как он умирает.

# Жизненный цикл MonoBehaviour. Порядок вызова Event функций в runtime режиме Unity.
- Awake() - Вызывается при загрузке скрипта и при активации объекта, до того как начнётся работа сцены. Для внутренней реализации
- OnEnable() - Вызывается каждый раз, когда объект или компонент становится активным.
- Start() - Вызывается один раз, перед первым кадром, но после Awake(). Для работы с другими объектами сцены
- Update() - Вызывается каждый кадр. Ввод, передвижение без физики
- FixedUpdate() - Вызывается с фиксированным интервалом времени (обычно для физического обновления).
- LateUpdate() - Вызывается после Update() каждого кадра. Камера, посткоррекция
- Рендеренги
- Пострендеренги.
- OnDisable() - Вызывается при деактивации объекта или компонента.
- OnDestroy() - Вызывается при уничтожении объекта или компонента.

# В чем проблемы Update?
- Каждый Update() вызывает Unity. Если у нас, например, 1000 MonoBehaviour, Unity каждый кадр должна вызвать каждый метод.
- Unity написана на C++, а твой код — на C#. Каждый Update() требует перехода.

    Когда таких вызовов тысячи, накладные расходы становятся заметными. (на C++ обработка быстрее, мультиплатформенность).
- Каждый кадр мы отрисовываем через Update. Сложнее контролировать порядок выполнения.
- Становится сложно понимать: кто раньше обновился; кто позже; кто изменил данные.

# Что такое Game Object? Что такое сцена?
GameObject - игровой объект, расположенный на сцене, к которому добавляются компоненты для определения поведения, внешнего вида и логики.

Если они не на сцене, они не gameObject’ы. Сцена – контейнер для GameObject-ов, то есть коллекция.

# Что такое префабы? В чем разница между префабом и сценой?
**Prefab** — это шаблон игрового объекта, который хранится как отдельный ассет в проекте.

Он содержит все компоненты, настройки и иерархию объекта.

Префабы позволяют многократно использовать один и тот же объект, а изменение исходного префаба автоматически может применяться ко всем его экземплярам на сцене.

**Scene** — это уровень или часть игры, которая содержит размещенные игровые объекты, освещение, камеры, UI и другие элементы.

В отличие от префаба, сцена хранит состояние всей игровой локации, а не одного объекта.

# Какие основные компоненты Unity?
- Transform — есть у каждого GameObject, отвечает за позицию, поворот и масштаб.
- Mesh Renderer и Mesh Filter — отображение 3D-моделей.

# Что такое Rigidbody?
Rigidbody — это компонент, который позволяет объекту подчиняться законам физики.

# Что такое Collider, какие типы Collider существуют? В чем разница между коллайдерами в режиме IsTriggered и обычном?
Он определяет геометрическую форму объекта для системы физики.

Типы: 
- Box, 
- Sphere, 
- Capsule, 
- Mesh
- Terrain

Если у Collider отключен IsTrigger, объект участвует в физических столкновениях и вызываются события OnCollisionEnter, OnCollisionStay и OnCollisionExit.

Если включить IsTrigger, физического столкновения не происходит — объекты могут проходить друг сквозь друга, но Unity вызывает события OnTriggerEnter, OnTriggerStay и OnTriggerExit.

# Знаете ли вы о Layer Collision Matrix и для чего она применяется?
Layer Collision Matrix — это настройка физики, которая позволяет определить, какие слои объектов могут сталкиваться друг с другом, а какие — нет.

Она находится в Project Settings → Physics

# Что такое deltaTime и fixedDeltaTime? Отличия между ними.
**deltaTime** – это время, которое проходит между кадрами. Это значение изменяется в зависимости от частоты кадров.

Используется в методе Update() для создания плавного движения (анимации), которые должны быть независимы от производительности системы.

**fixedDeltaTime** – время для физических расчетов и симуляций, интервал времени между вызовами FixedUpdate() (обычно 0.02 секунды по умолчанию).

# Как вы реализуете системы физики в Unity? Расскажите о вашем подходе к моделированию физических взаимодействий объектов в игре.

Для реализации физики в Unity я сначала определяю, должна ли система быть полностью физической или управляемой кодом.

Если объект должен реагировать на силы, гравитацию и столкновения, использую Rigidbody и Collider.

Если нужна более предсказуемая логика персонажа, могу использовать CharacterController или кинематический Rigidbody.

# Расскажите о вашем опыте в разработке искусственного интеллекта (AI) в играх с использованием Unity. Какие алгоритмы вы используете для моделирования поведения NPC (неконтролируемых персонажей)?
Для перемещения использовал NavMeshAgent, а поведение строил на основе конечного автомата состояний (Finite State Machine).

В зависимости от ситуации NPC переключался между состояниями, например: Patrol, Chase, Attack и Return.

FSM проста в реализации, хорошо читается и отлично подходит для большинства NPC с небольшим количеством состояний.

# Опыт работы с UI компонентами? Что такое канвас? Что такое панель? Чем плох и хорош канвас? Как верстать адаптивный интерфейс? Что такое LayoutGroup?
**Canvas** – это холст, на котором мы размещаем UI элементы. В играх иногда использую несколько Canvas.

**Panel** обычно используется как контейнер для группировки элементов.

**Плюсы Canvas:** легко менять размер и масштаб. Можно собрать все UI элементы в одном месте.

**Минусы Canvas:** пересчитывает и перерисовывает всё. Например, если изменяется текст, изображение, позиция, при анимациях (Animator).

Производится расчет их позиций, размеров и внешнего вида.

Для адаптивной верстки применяю Canvas Scaler в режиме Scale With Screen Size, правильно настраиваю Anchors и использую Layout Group для автоматического расположения элементов.

**Layout Group** перегружены, потому что они меняют transform.position конкретного UI элемента каждый кадр.

# Занимались ли вы версткой UI? Какие знаете проблемы в работе с этим? Какие знаете подходы к оптимизации UI?
Делал игровые HUD, меню, окна магазина, настройки, экраны победы и поражения, различные popup-окна. 

Для этого использовал Canvas, RectTransform, Anchors, Canvas Scaler, Layout Group, Content Size Fitter, Scroll View и TextMeshPro.

Основная задача была сделать интерфейс корректно отображающимся на разных разрешениях мобильных устройств.

**Подходы по оптимизации UI:**
- Разделять UI на несколько Canvas
- Отключать Raycast Target там, где он не нужен
- Не использовать LayoutGroup без необходимости
- Избегать частого изменения текста
- Минимизировать Overdraw - не накладывать много полупрозрачных Image друг на друга
- Использовать Object Pool для UI-элементов
- Отключать неиспользуемые окна

# Расскажите, какие лучшие практики при работе с UI вы знаете?
- Использовать MVP/MVVM-подход - UI View только отображает данные и обрабатывает клики, а логика находится отдельно.
- Правильно настраивать Anchors, Pivot и Canvas Scaler 
- Не делать один огромный Canvas 
- Отключать лишний Raycast Target 
- Использовать Object Pool 
- Не обновлять UI каждый кадр без необходимости - Например, счетчик монет обновлять только при изменении значения, а не в Update().

# Какой архитектурный паттерн наиболее подходит для реализации пользовательского интерфейса в Unity? Почему?
Для пользовательского интерфейса чаще всего выбрал MVP — Model–View–Presenter.

View отвечает только за отображение и пользовательский ввод, Model хранит данные и бизнес-логику, а Presenter связывает их между собой.

MVC тоже можно применять, но граница между View и Controller часто получается менее очевидной, потому что MonoBehaviour одновременно принимает события интерфейса.
MVVM подходит для UI с привязкой данных:
- ViewModel предоставляет свойства и команды; 
- View обновляется через data binding; 
- меньше ручных вызовов вроде SetBalance().

# Аниматор. Можно ли дописывать логику к состояниям аниматора? Что такое Timeline и опыт работы с ним?
**Animator** - управление сменой анимаций у объекта.

Можно добавить собственную логику двумя способами:
- State Machine Behaviours - специальные скрипты, которые можно привязать к состояниям аниматора. Они позволяют запускать код, когда персонаж входит в состояние (OnStateEnter), обновляется в нём (OnStateUpdate) и выходит из него (OnStateExit).
- Animation Events - Позволяют вызывать методы на объекте в определённый момент анимации.

**Timeline** — это инструмент для создания временных последовательностей.
С его помощью можно синхронизировать анимацию, аудио и даже движение камеры, чтобы создавать кат-сцены и сложные анимационные переходы.

# Камера. Типы камер, параметры для настройки. Скай бокс, occlusion culling.
Камера является центральным компонентом для визуализации сцены.
- Перспективная: Объекты дальше кажутся меньше – используется в 3D-играх для создания глубины. 
- Ортографическая: Объекты сохраняют одинаковый размер независимо от расстояния (2D или изометрических играх)
- Field of View (FOV): Угол обзора для перспективной камеры – чем больше, тем шире охват, но возможны искажения. 
- Orthographic Size: Размер видимой области для ортографической камеры. 
- Clipping Planes: Минимальное и максимальное расстояние, в пределах которого объекты видны. 
- Depth и Clear Flags: Определяют порядок отрисовки и способ очистки экрана перед рендерингом. 
- Viewport Rect: Область экрана, где показывается изображение – полезно для разделения экрана. 
- SkyBox - это фон, создающий иллюзию окружающего мира (неба, облаков и т.д.), чтобы игра выглядела живой. 
- Occlusion Culling - эта функция помогает игре не рисовать объекты, которые не видны камере, тем самым ускоряя работу игры.

# Что такое mesh? Из чего состоит 3д модель?
**Mesh** – это набор вершин (точек), между которыми проведены линии (ребра), в результате формируется Mesh (полигон). Mesh состоит из треугольников и квадратов.
- Вершины (точки) - они определяют форму объекта в пространстве. 
- Грани (лицевые поверхности) - они создают поверхность модели. 
- Нормали - они помогают понять, как свет падает на объект. 
- UV-координаты - UV-координаты говорят, как правильно расположить текстуру на модели. 
- Материалы и текстуры. 
- Скелет (Rigging). 
- Анимации 
- Коллайдеры

# Какие данные в себе хранит меш (Mesh)?
- Vertices (Вершины)
- Triangles (Индексы). Mesh не хранит сами треугольники. Он хранит индексы вершин, из которых они состоят. 
- Normals - Направление поверхности.
- UV Coordinates - Координаты текстуры. 
- Tangents - Используются для Normal Map и некоторых шейдеров. 
- Vertex Colors - Цвет каждой вершины.

**Mesh** — хранит геометрию.

**MeshRenderer** — отображает эту геометрию на экране.

**MeshFilter** - хранит ссылку на Mesh.

**Например**, при работе с шейдерами мне приходилось учитывать UV-координаты и нормали, так как именно они используются для корректного наложения текстур, расчета освещения и работы таких эффектов, как обводка объектов.

# Опыт работы с шейдерами. Приходилось ли писать шейдеры?
**Шейдеры (Shader)** используются для создания различных визуальных эффектов. Они могут изменять внешний вид материалов и позволяют создавать реалистичное освещение, прозрачность, карты нормалей, рельефы и многое другое.

**Shader Graph** – это визуальный инструмент, который позволяет создавать шейдеры с помощью системы нодов.

**Например**, делал шейдер обводки интерактивных объектов, чтобы подсвечивать их при наведении, а также шейдер тумана — это был один из самых сложных для меня. 

В нем нужно было рассчитывать интенсивность тумана в зависимости от расстояния до камеры, плавно смешивать цвет тумана с цветом объекта и добиться естественного визуального эффекта без резких переходов.

При этом приходилось учитывать производительность, чтобы эффект не создавал лишнюю нагрузку на мобильных устройствах.

**Материал** задаёт внешний вид объекта: его цвет, текстуру и прочие визуальные эффекты. Обертка над шейдером

# Батчинг (Batching) и Draw calls. Что это? Какие подходы оптимизации вызовов отрисовки вы знаете?
**Draw call** – это количество вызов отрисовки.

**Батчинг** – это способ группировать объекты так, чтобы их можно было нарисовать за один вызов.
- Static Batching - Объединяет неподвижные объекты в один блок, рисуя их за один вызов. 
- Dynamic Batching - Группирует движущиеся объекты (у них одинаковый материал) для уменьшения количества вызовов. 
- GPU Instancing - Позволяет рисовать множество копий одного объекта за один вызов, даже если они выглядят по-разному (MaterialPropertyBlock)
  - MaterialPropertyBlock (проперти блоки) – позволяет изменять параметры материала для отдельных объектов без изменения самого материала. Вместо изменения самого материала, мы создаем проперти блок, который задает нужные параметры. 
- Объединение материалов и использование атласов текстур - Снижают количество переключений между мат-ми и тек-ми. 
- Occlusion Culling - Не отрисовываются объекты, которые не видны с текущей позиции камеры. 
- Level of Detail (LOD) - Используются модели с меньшим количеством полигонов для объектов, находящихся далеко от камеры.

# Как определить и какие действия нужны, чтобы исправить утечку памяти (Memory Leak)?
Сначала воспроизвожу сценарий несколько раз: например, открываю и закрываю окно, загружаю и выгружаю сцену, создаю и удаляю объекты. 

Если после каждого цикла память растёт и не возвращается к примерно исходному уровню, это признак возможной утечки.

**Для диагностики использую:**
- Unity Profiler → Memory — смотрю общий рост памяти и количество объектов; 
- Memory Profiler — делаю snapshot до и после проблемного сценария и сравниваю их; 
- Profiler на реальном устройстве, потому что на мобильном утечки могут проявляться иначе.

**В первую очередь проверяю:**
- Подписки на события

Частая причина — объект уничтожен, но остался подписан на событие долгоживущего сервиса. 
- Статические ссылки и Singleton

Статическое поле может удерживать объект после смены сцены. Нужно очищать такие ссылки, когда объект больше не используется. 
- Корутины, Task и CancellationToken

Асинхронная операция может продолжать удерживать MonoBehaviour, замыкание или другие данные после уничтожения объекта. Поэтому долгие операции нужно отменять при OnDestroy. 
- Addressables и загруженные ресурсы

После загрузки ресурса нужно корректно освобождать handle.
Для сцен — использовать соответствующую выгрузку. Иначе ассеты и их зависимости могут оставаться в памяти. 
- Object Pool

Пул сам по себе не является утечкой, но если он бесконтрольно растёт и никогда не уменьшает размер, память тоже будет постоянно увеличиваться. Нужно ограничивать максимальный размер пула.

# Какие типы рендеринга вы знаете? Каковы их различия?
Render Pipeline преобразует данные сцены в итоговое изображение на экране.

**3 основных Render Pipeline:**
- Built-in,
- Universal (URP)
- High Definition (HDRP).
- 
Они отличаются возможностями, качеством графики и производительностью.

**URP:**
- высокая производительность;
- поддерживает Shader Graph;
- имеет современные возможности освещения и постобработки.

# Как реализовать систему, которая позволяла бы добавлять в игру контент без обновления приложения? Какие механизмы для этого можно использовать?
Такую систему можно реализовать через удаленную загрузку контента. 

Обычно для этого используют Addressables или AssetBundles. 

Идея в том, что базовое приложение содержит только основной код и минимальные ресурсы, а новый контент — сцены, префабы, текстуры, конфиги, иконки — загружается с сервера по мере необходимости.

Например, если нужно добавить новый ивент, можно хранить его конфигурацию на сервере в JSON или Config, а визуальные ресурсы — в Addressables. 

При запуске игра проверяет актуальный каталог Addressables, скачивает новые AssetBundles, загружает нужные префабы и данные, после чего контент становится доступен игроку без обновления приложения в сторе.

# Какова ситуация с асинхронностью в Unity? Какие есть подводные камни?
Unity поддерживает несколько способов выполнения асинхронных операций: корутины (Coroutine), стандартные async/await и сторонние библиотеки, например UniTask.

В своих проектах, в большинстве случаев, я использовал корутины.

Главный подводный камень заключается в том, что большинство не потокобезопасно. 

Обращаться к GameObject, Transform, MonoBehaviour или другим объектам Unity можно только из главного потока.

Поэтому тяжелые вычисления можно выполнять в фоне, но после их завершения возвращаться в главный поток, если нужно изменить состояние объектов сцены.

# Как вы работаете с коллизиями в Unity? Как обнаруживать столкновения между объектами и как избежать физических проблем, таких как проникание (clipping) или неправильная обработка столкновений?
Для обнаружения столкновений в Unity использую Collider вместе с Rigidbody. 

Для физических столкновений применяю OnCollisionEnter, OnCollisionStay и OnCollisionExit, а для зон, которые не должны физически останавливать объект, включаю Is Trigger и использую OnTriggerEnter, OnTriggerStay и OnTriggerExit.

Для большинства событий хотя бы один из взаимодействующих объектов должен иметь Rigidbody.

Чтобы избежать проникания объектов, не перемещаю Rigidbody напрямую через Transform.

Для физического движения использую MovePosition, velocity или AddForce в FixedUpdate.

Для быстрых объектов, например снарядов, включаю Continuous или Continuous Dynamic Collision Detection, потому что объект может пролететь сквозь тонкий коллайдер между физическими кадрами.

# Ваши действия для локализации проблемы просадки FPS на мобильном устройстве? Если в редакторе проблема не воспроизводится?
Если проблема проявляется только на мобильном устройстве, то анализирую ее именно на устройстве, а не в редакторе.

- Сначала пытаюсь понять, в какой момент происходит просадка: при загрузке сцены, в бою, во время появления большого количества объектов или при открытии UI. 
- Подключаю Unity Profiler 
- Определяю, где узкое место:
  - если загружен CPU — ищу тяжелые методы, частые вызовы Update, лишние аллокации;
  - если проблема в GPU — проверяю Draw Calls, Overdraw, количество полигонов, постобработку, шейдеры; 
  - если вижу частые GC — ищу лишние выделения памяти.
- Использую Frame Debugger, чтобы посмотреть количество Draw Calls, батчинг и лишние проходы рендеринга.
- На Android дополнительно смотрю Logcat, чтобы исключить ошибки SDK, проблемы с памятью или другие сообщения системы.

# Какие навыки и практики вы используете для поддержки кодирования игры в Unity большими командами разработчиков?
При работе в команде я стараюсь придерживаться единых стандартов кодирования и делать код максимально понятным и предсказуемым для других разработчиков.

Для этого использую модульную архитектуру, соблюдаю принципы SOLID, разделяю ответственность между классами и избегаю сильной связанности.

Также использую Git с GitFlow, создаю небольшие feature-ветки, регулярно делаю коммиты с понятными сообщениями и участвую в code review.

При разработке стараюсь не создавать большие классы, использовать понятные имена и документировать нетривиальные решения, чтобы код было проще поддерживать всей команде.
